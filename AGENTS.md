  - **`dev-start.ps1` (and `dev-start.bat`) invokes its child PowerShell scripts via `powershell.exe` (Windows PowerShell 5.1), NOT `pwsh.exe` (PowerShell 7+).** Any fix targeting a dev-start child script (`scripts/render-mcp-config.ps1`, `scripts/op-mcp-doctor.ps1`, `.cursor/hooks/sync-types.ps1`, etc.) must remain compatible with 5.1 syntax and behavior. **`dev-start.ps1` itself must stay pure ASCII** — the file is saved UTF-8 *without* BOM, and PowerShell 5.1 reads BOM-less files as Windows-1252, so a single em-dash (`U+2014`, bytes `0xE2 0x80 0x94`) gets reinterpreted such that its `0x94` trailing byte acts as a smart closing quote and silently terminates a string, producing cascading bogus parse errors hundreds of lines later (`&&` "not a valid statement separator", parens in comments triggering "expression expected", etc.). Validate with `powershell.exe -NoProfile` (5.1 explicitly), not `pwsh.exe`, and rewrite the file with `[System.Text.ASCIIEncoding]` if any non-ASCII byte is introduced. Same constraint applies to other 5.1-targeted scripts.
  - **Prefer in-process `& script.ps1` invocation over `& powershell.exe -NoProfile -ExecutionPolicy Bypass -File script.ps1`** when a child PowerShell script needs to share env vars (especially `OP_SESSION_*` from 1Password). Spawning a fresh `powershell.exe` child gives that child its own environment and forces a second `op` auth handshake (and an extra biometric/desktop-app prompt for non-service-account users). `dev-start.ps1`'s 1Password env-sync + MCP-render steps are wired this way; if you reintroduce the spawn pattern there, the duplicate-prompt regression returns. `&`-invoked scripts run in a child variable scope so they can't pollute the parent, and `exit N` from inside them only sets `$LASTEXITCODE` (does NOT terminate the parent) — bracket the call with `$ErrorActionPreference = 'Continue'` + a `finally` restore if the parent uses `Stop` mode.
  - **`gcloud` on Windows is itself a PowerShell wrapper at `gcloud.ps1` that invokes `python.exe`** — the deepest layer of the stderr-as-terminating-error issue described above.
  - **`$var:` inside double-quoted PowerShell strings is parsed as a scope qualifier** (like `$env:` or `$global:`), not as a variable followed by a colon — this throws a parser error before the script even runs. When emitting log lines like `"$var: missing"`, always use `${var}:` to delimit the variable name from the trailing colon. `scripts/sync-vercel-from-1password.ps1` previously broke on exactly this.
  - **Native-command splat traps on PS 5.1.** `'npx vercel'.Split() @args` does NOT splat the way it does in PS 7 — the whole `'npx vercel'` string is passed as the command name and the call fails. Use the call operator with explicit pieces: `& npx vercel @args` or build the full argument array and invoke `& npx @argArray`. Similarly, passing a switch with an empty-string value (e.g. `--token $env:VERCEL_TOKEN` when `VERCEL_TOKEN` is unset) makes the parser consume the next positional argument as the value, silently corrupting the call — guard with a conditional that omits the flag entirely when its value is empty (`if ($env:VERCEL_TOKEN) { $cmdArgs += @('--token', $env:VERCEL_TOKEN) }`).
  - **Literal `${{` in PowerShell strings is parsed as a variable reference** and breaks `git commit -m "...${{ secrets.X }}..."` invocations before git ever runs. When a commit message, PR body, or workflow snippet must contain `${{`, write the body to a temp file and pass it via `--file` / `--body-file`, or wrap the literal in a single-quoted here-string.
  - **Bash heredoc syntax (`<<EOF`, `$(cat <<'EOF' ... EOF)`) does not work in PowerShell** — the parser treats it as an unterminated string and the call fails before git/gh ever runs. For multi-line commit messages, PR bodies, or issue bodies on this Windows host, write the body to a temp file (e.g. under `$env:TEMP`) and pass it via `git commit -F <file>`, `gh pr create --body-file <file>`, or `gh issue create --body-file <file>`. Reserve inline `-m "..."` / `--body "..."` for short single-line strings that contain no `${{`, no newlines, and no embedded double-quote characters (inner `"` prematurely terminates the outer quoted string on the command line, and the remaining text is parsed as positional CLI arguments by `gh`, producing "unknown arguments" errors). The PowerShell single-quoted here-string (`$body = @'...'@; --body $body`) is a viable middle-ground for multi-line strings that contain no embedded double quotes — but when in doubt, use `--body-file`.
  - **New `.sh` scripts authored from PowerShell need explicit LF normalization and the git executable bit before commit.** The `Write` tool / `Set-Content` / `Out-File` defaults emit CRLF, which `bash -n` and the GitHub Actions Linux runners cannot parse (the `\r` becomes part of the last token of every line and shells fail with cryptic `unexpected end of file` / `command not found` errors). After writing a new `*.sh`, normalize via `[System.IO.File]::WriteAllText((Resolve-Path $p), ((Get-Content -Raw $p) -replace "`r`n","`n"), [System.Text.UTF8Encoding]::new($false))`, then `git update-index --chmod=+x <path>` before staging — the repo's `.gitattributes` does not currently force LF or `text eol=lf` on `.sh` files. Verify the staged blob is CRLF-free with `git cat-file -p <hash>` (or a one-line Python check on the blob bytes) before pushing. `scripts/ci/supabase-secrets-drift-preflight.sh` is the canonical first instance and the working reference for new CI shell scripts.

- **Local Supabase port conflicts on Windows are an OS-level reservation issue, NOT a Docker leak.** Supabase local ports are fixed (`54321` API, `54322` Postgres, `54323` Studio, `54324` Inbucket, `54327` Analytics, `54328` Pooler); never change them on the fly to bypass a conflict — that breaks every other tool that hardcodes the canonical ports. The Windows Docker error `bind: An attempt was made to access a socket in a way forbidden by its access permissions` (Winsock `WSAEACCES`) on port `54322` (or any 543xx port) means **the Windows kernel itself has reserved that port chunk**, NOT that any process is listening. Root cause: Supabase's port band sits inside Windows' default ephemeral port range (49152-65535), and WinNAT / Hyper-V dynamically allocate 100-port chunks within that range for outbound NAT mappings (Hyper-V VMs, WSL2 distros, VPN clients). When one of those chunks lands on top of 54321-54328, every `bind()` against the trapped port fails. Diagnose with `Get-NetTCPConnection -LocalPort 54322 -State Listen` (expect empty — no user-space process), `docker ps -a` (expect zero containers — Docker is not the cause), and `netsh int ipv4 show excludedportrange protocol=tcp` (look for a non-`*`-flagged range that overlaps the Supabase ports). **`wsl --terminate docker-desktop` does NOT fix this** — verified empirically; the reservation belongs to the Windows TCP/IP stack, not Docker, and persists across full Docker Desktop shutdowns. The fix is a one-time persistent administered exclusion: `.\scripts\reserve-supabase-ports.ps1` (auto-elevates) registers ports 54321-54328 as "administered" (asterisk-flagged) excluded ports via `netsh int ipv4 add excludedportrange ... store=persistent`, after which the Windows kernel will never auto-allocate them again across reboots, Docker restarts, or WSL2 churn. The helper requires `net stop winnat` first (briefly, ~2s window) because Windows refuses to add an administered exclusion that overlaps an active dynamic reservation, and the misleading error it returns is `ERROR_SHARING_VIOLATION` ("the process cannot access the file because it is being used by another process") which has nothing to do with file access. `dev-start.ps1` and `dev-stop.ps1` only **diagnose** this trap (scan `excludedportrange` after the existing port sweep) and print the exact one-time-fix command — they do NOT auto-elevate or auto-fix, because (a) it would prompt UAC mid-flow and (b) it couples a per-session dev script to a one-time-per-machine setup task. See `.cursor/rules/local-dev-troubleshoot.mdc` failure mode #6 for full diagnostic details.

- **Migration idempotency rules — corollary to the migration freeze rule above.** (1) `ALTER FUNCTION ... SET search_path = public, pg_temp` (and any other bare `ALTER` against a function) MUST be wrapped in a `DO $$ BEGIN ... IF EXISTS (SELECT 1 FROM pg_proc p JOIN pg_namespace n ON n.oid = p.pronamespace WHERE n.nspname = 'public' AND p.proname = '<fn>' AND pg_catalog.pg_get_function_identity_arguments(p.oid) = '<exact arg list>') THEN ALTER ...; END IF; END; $$;` block — otherwise the migration fails with `function ... does not exist` on any environment that ran a different migration subset (fresh local Supabase, partial rollback, signature drift). Use `pg_get_function_identity_arguments(p.oid)` for the signature comparison, NOT raw OID arrays — it strips DEFAULT clauses and matches the canonical Postgres rendering, so a function declared with `p_actor_id UUID DEFAULT NULL` still matches a literal `p_actor_id uuid` arg-list string. (2) The `DO $guard$ ... IF (object_exists) THEN skip ELSE apply $guard$;` defensive pattern in an EARLIER-timestamped migration that anticipates a LATER migration creating an object can backfire when the later migration is edited in-place AFTER its timestamp has already landed in `schema_migrations` — Supabase will not re-run the now-edited later migration, so the guard's "object exists" branch fires and skips, but the later migration's in-place edits never deliver either. Net effect: neither migration applies the intended fix on already-deployed environments (e.g. preview). Resolution: ship a NEW later-timestamped migration (e.g. `20260426000000_fix_<thing>_canonical.sql`) that unconditionally calls `CREATE OR REPLACE FUNCTION` and branches on the dependent object's existence at runtime to pick the right body, so the fix lands regardless of which intermediate state the environment is in. (3) **Multiple migrations that `CREATE OR REPLACE FUNCTION` against the same Postgres function silently fight: the latest-timestamped migration wins and overwrites every earlier in-place body fix.** Cursor Bugbot caught exactly this on PR #670 (round 3) — an overdue-filter fix shipped in `20260424000100_work_order_overdue_history.sql` was silently reverted by `20260425000000_implement_equipment_status_history.sql` because the later migration also `CREATE OR REPLACE`d `get_dashboard_trends` with the pre-fix body (the function rewrite was incidental to the later migration's stated purpose, just bundled in as part of an RPC update). When shipping any in-place RPC body fix in migration N, audit EVERY later-timestamped file in `supabase/migrations/` that also defines the same function (`rg 'CREATE OR REPLACE FUNCTION public\.<fn>' supabase/migrations/`) and forward-port the fix into ALL of them — including the `COMMENT ON FUNCTION` so the documentation thread is preserved on every environment regardless of which subset of migrations has applied. The same overwrite trap applies to `CREATE OR REPLACE TRIGGER`, `CREATE OR REPLACE VIEW`, `CREATE OR REPLACE PROCEDURE`, `CREATE OR REPLACE RULE`, and any `COMMENT ON ...` chain. (4) Production migration drift can be a timestamp mismatch rather than missing SQL: if production has a migration version whose name already exists locally under a different timestamp and the SQL matches, rename/repair the local file to the production timestamp instead of adding a duplicate migration file.
